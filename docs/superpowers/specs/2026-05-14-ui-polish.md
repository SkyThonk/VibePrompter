# UI/UX Polish & Native Windows Experience — Spec

**Date:** 2026-05-14
**Loop:** Ralph (iterative polish — re-fired until quality bar met)

## Goal

Transform the existing Tauri + React desktop app into a polished, production-grade Windows
desktop experience that feels native, lightweight, modern, and premium. The app already has
backend architecture, AI integration, shortcuts, tray, overlays, settings, modes, and a
command palette. **Goal is NOT to rebuild functionality** — it is to refine UX, UI
consistency, animations, layout, native desktop behavior, interaction quality,
accessibility, visual hierarchy, and performance perception.

## Target feel

* Raycast for Windows
* Linear-quality UX
* Native Windows 11 utility
* AI-first productivity software

The application MUST NOT feel like: a wrapped website, an admin dashboard, a bulky
Electron app, a generic Tailwind template.

## Primary UX adjectives

instant • calm • intelligent • keyboard-first • minimal • premium • focused •
invisible-until-needed

## Windows 11 design language

mica-inspired surfaces • acrylic blur • subtle transparency • rounded corners • soft
elevation • minimal borders • smooth transitions • adaptive dark mode. Avoid: excessive
shadows, loud gradients, bright colors, oversized controls, gaming aesthetics.

## Visual style

Restrained palette: charcoal / graphite / muted slate / soft neutral grays. Accents: soft
blue, muted cyan, subtle purple. Color used only to indicate focus, actions, states.

Typography: Inter or Geist; compact, elegant, highly readable. Hierarchy: large clean
titles, muted secondary labels, compact metadata, subtle helper text.

## Component quality

* **Buttons** — tactile, subtle hover, polished active. Primary: slightly elevated, stronger contrast. Secondary: ghost/minimal. Danger: muted red.
* **Inputs** — resemble command-palette inputs, soft borders, elegant focus glow, compact, keyboard nav, smooth focus transitions, instant validation.
* **Cards & panels** — soft transparency, layered depth, subtle blur, minimal borders. Avoid heavy boxed layouts.

## Command palette (most important)

instantaneous, buttery smooth, premium, distraction-free. Smooth open/close (fade + slight
scale), instant keyboard focus, zero layout shift, fuzzy search, keyboard-only navigation,
compact action rows (icon, title, optional subtitle, keyboard hint), fluid hover states.

## Overlay windows

Appear near cursor intelligently, animate smoothly, avoid edge clipping, multi-monitor
support. Animations: soft fade, slight upward movement, fast response. Never jarring.

## System tray

Native context-menu styling, instant responsiveness, clean status indicators, subtle app
states, Windows 11 density.

## Settings window

Feel like Linear / VS Code / modern native tools — NOT enterprise admin dashboards. Compact
sidebar, smooth section transitions, persistent navigation, instant save feedback,
searchable settings.

## Prompt mode manager

Users feel powerful, in control, creative. Editor like modern IDE / AI playground /
structured prompt engineering. Syntax highlighting, templates, live preview, token
estimation, provider selection, test execution.

## History

Lightweight, useful, non-intrusive. Compact cards, expandable previews, subtle metadata,
clean timestamps. Quick reuse, copy result, restore transformation.

## Animation

Duration 120–220ms. Preferred motions: opacity fade, subtle scale, small translations.
Avoid bounce/elastic/exaggerated. Respect `prefers-reduced-motion`.

## Microinteractions

Hover glow, button press depth, smooth toggles, animated loading, progressive AI streaming
(intelligent / alive / responsive typing indicators).

## Performance perception

App must FEEL fast even during AI latency. Optimistic UI, skeleton states, streaming
responses, animated placeholders. Never freeze UI.

## Empty states

Elegant: minimal illustrations, helpful onboarding text, actionable buttons. Cases: no
history, no providers, no modes configured.

## Errors

Calm, helpful, recoverable. Never expose raw technical errors. Friendly explanation, retry
actions, diagnostics on demand.

## Keyboard-first

Every important interaction supports keyboard nav, command shortcuts, focus management,
accessibility. Most features operable without mouse.

## Accessibility

Screen readers, high contrast, reduced motion, scalable text, visible focus states.
Native, not bolted on.

## Responsive desktop

Ultrawide, small laptop, multi-monitor, DPI scaling. Windows remember size/position, open
intelligently, preserve workflow continuity.

## Loading states

Smooth indicators, shimmer, streaming feedback, progressive rendering. Never blocking
spinners unless unavoidable.

## Design consistency

Centralized design-token system. Consistent spacing, radius, iconography, predictable
layouts, coherent rhythm.

## Recommended UI stack

React, TailwindCSS, Framer Motion, shadcn/ui. Additions: Radix UI, cmdk, lucide-react.

## Windows-specific polish

Native titlebar integration, blur/transparency, startup animations, tray-native behavior,
proper DPI awareness, dark mode sync with OS, native window shadows.

## UX goal

User feels: *"This application belongs on my operating system."* Invisible when idle,
powerful when invoked, extremely responsive, thoughtfully designed.

## Final quality bar

Resemble: Raycast, Linear, Cursor, Arc Browser, modern Windows utilities.
NOT: bootstrap dashboards, generic admin panels, hobby apps. Commercially production-ready.

## Iteration log

Tracked at the bottom of this file. Each Ralph loop iteration appends one entry
describing what changed and what's next.

### Iter 1 — Theme attribute fix + backend bridge

**Problem found:** `ThemeProvider` set `<html class="light|dark">`, but the design-token
CSS (`tokens.css`) keys off `[data-theme="dark"]` / `[data-theme="light"]`. Result: light
theme didn't actually apply, and the backend `theme` / `accent` settings in the Appearance
panel didn't reach the rendered UI.

**Fix:**
1. `ThemeProvider` now writes `data-theme` (and `data-accent`) attributes on `<html>`
   instead of class names.
2. Added a `BackendThemeBridge` component that reads `useAppSettingsQuery` and pushes
   the backend `theme` and `accent` settings into the ThemeProvider whenever they
   change — so toggling theme/accent in the Appearance panel now actually re-paints
   the app.
3. Theme `system` option still respects OS dark-mode preference and listens for
   changes.

**Next likely iteration target:** Command palette polish (animation, keyboard nav,
hover states) — the spec calls it the most important UI element.

### Iter 2 — Command palette: keyboard nav, mount animation, scored fuzzy match

**Problems found:**
1. The palette advertised ↑/↓ keyboard navigation in the section header, but the
   keydown handler only caught `Enter` and `Escape`. Arrow keys did nothing.
2. `Enter` always set `state='loading'` regardless of which action was focused —
   the focused action's identity was never read.
3. No entry animation; the palette popped in without easing.
4. `filterActions` did substring matching only; results weren't ranked, so a
   match in `hint` outranked a label-prefix match if it came first in the seed.
5. `prefers-reduced-motion` was not honored anywhere globally.

**Fix:**
1. `CommandPalette.tsx` — Arrow Up/Down (wrap), Home/End, and Enter now navigate
   and trigger the focused action. Focus resets to 0 when the filter changes;
   the focused row is scrolled into view on each change. The action grid carries
   `role="listbox"` + per-row `role="option"` / `aria-selected` so screen readers
   announce the active item.
2. New `triggerAction(index)` helper centralizes the "fire the focused match"
   path; an empty filter still allows Enter-to-submit-as-custom-instruction.
3. `ActionRow` now accepts an `onClick` so the mouse path mirrors the keyboard path.
4. `filterActions.ts` rewritten as a small scored matcher: label-prefix (100) >
   label-contains (70) > id-prefix (60) > id-contains (40) > hint-contains (30)
   > subsequence-in-label (20). Stable secondary sort by original index.
5. `tokens.css` — added `ph-anim-pop-in` keyframes (fade + 1.5% scale + 2px lift
   over 180ms with a calm easing) and applied it to the palette card on mount.
6. `tokens.css` — added a global `@media (prefers-reduced-motion: reduce)` rule
   that clamps animation/transition durations to ~0ms across the whole app, so
   every animation added in future iterations inherits accessibility for free.

**Next likely iteration target:** Settings window — section transitions, search,
and instant save feedback; Linear/VS-Code feel instead of admin-dashboard feel.

### Iter 3 — Settings window: working search, section fade, live save state

**Problems found:**
1. The "Search settings…" input was purely decorative — no `value`, no `onChange`,
   no filtering. Users typed and nothing happened.
2. Tabs hard-switched with no transition — felt like an admin dashboard.
3. The spec calls for "instant save feedback" but nothing in the UI acknowledged
   that a `save_settings` mutation had fired or completed. Saves were invisible.

**Fix:**
1. `SettingsWindow.tsx` — search is now real: filters the tab list by label /
   id, shows "No matches" when empty, supports Enter (jump to first match) and
   Escape (clear). Added `aria-label="Settings sections"` on the nav.
2. Added a soft 160ms `ph-fade-in` (opacity + 3px upward translate) on the main
   content area, retriggered via `key={tab}` on the `<main>` element. Tabs
   now glide in instead of snapping.
3. New `SaveIndicator` lives in the sidebar footer. Reads `useIsMutating()` —
   while any mutation is in flight, shows a pulsing accent dot with "Saving…".
   On completion it briefly shows an `ok` dot with "Saved" (1.4s) then fades
   out. Uses `useRef` + `useEffect` to detect the saving→idle transition
   without React anti-patterns.
4. `tokens.css` — added a second animation primitive `ph-anim-fade-in`
   (opacity + 3px translate, 160ms) alongside the heavier `ph-anim-pop-in`.
   Section-level transitions use this lighter motion; overlays/palette keep
   the scale-ier pop-in. Both honor the global reduced-motion clamp from iter 2.

**Next likely iteration target:** Empty states (no history, no providers, no
modes configured) — the spec calls for elegant minimal illustrations + helpful
onboarding text. Currently the History/Providers/Modes panels render nothing
useful when their lists are empty.

### Iter 4 — Elegant empty states

**Problems found:**
1. `HistoryPanel`, `ProvidersPanel`, and `ModesPanel` each had `if (!current) return null;`
   when their data was empty — a blank white area with no explanation, no
   onboarding, no call-to-action.
2. During the initial query (data still undefined), the same blank path triggered
   — empty state would briefly flash if the panels had any onboarding UI.
3. The History panel's "no matches for query X" was a single line of muted text
   — usable but graceless compared to the rest of the design.

**Fix:**
1. New reusable `EmptyState` component in `@shared/ui` — centered icon in a
   rounded subtle-bordered surface, large title, muted description, optional
   action row. Animates in via `ph-anim-fade-in` so it doesn't snap. Supports
   a `compact` variant for inline use.
2. `HistoryPanel`: empty → "No transformations yet" with the global shortcut
   chip in the description and a "Learn about shortcuts" CTA. The in-list
   no-search-matches case is now a `compact` EmptyState with a "Clear search"
   button.
3. `ProvidersPanel`: empty → "No providers configured" + "Add provider" CTA,
   emphasizing local-encrypted-storage trust.
4. `ModesPanel`: empty → "No prompt modes yet" + "Create your first mode" and
   "Import" CTAs.
5. All three panels now respect `isLoading` from TanStack Query — they render
   the panel head only while the first fetch is in flight (the panel-level
   fade-in from iter 3 covers the visual gap), so the empty state never flashes
   incorrectly.
6. The `head` JSX is now a captured variable so loading / empty / loaded
   branches share the same panel header without duplicating prop trees.

**Next likely iteration target:** Toggle, Slider, and PhButton microinteractions
— hover glow, press depth, smooth transitions. The spec calls for tactile
button feel + smooth toggle transitions, and these primitives are used
everywhere so the polish compounds.

### Iter 5 — Tactile primitives: PhButton press, Toggle compression, Slider hover/active

**Problems found:**
1. `PhButton` had hover state but **no press depth** — the spec explicitly calls
   for "tactile feel, button press depth". Buttons felt flat on click.
2. `PhButton`'s `transition` list didn't include `transform`, so any future
   press animation wouldn't have animated smoothly.
3. `Toggle` had a spring on the thumb glide but **no press feedback**. No way
   to feel the click landed. Also no `role="switch"` / `aria-checked` — screen
   readers couldn't announce its state.
4. `Slider` had no hover affordance on the thumb. The 14px thumb sat unchanged
   regardless of mouse proximity — felt static.
5. None of the three primitives supported a `disabled` state visually or for
   pointer events.

**Fix:**
1. `PhButton`:
   - Press depth via Tailwind `active:translate-y-px` + `active:duration-75`
     so the button drops 1px instantly and settles on release with the
     normal 150ms ease.
   - Widened transition list to `[background,color,box-shadow,transform]`.
   - Added `disabled` support — `opacity-50`, `cursor-not-allowed`, no
     active translate when disabled.
2. `Toggle`:
   - Added `role="switch"` + `aria-checked` + optional `aria-label` so
     screen readers announce on/off correctly.
   - Press feedback: thumb scales to 0.86× on active (90ms) via a CSS rule
     in `tokens.css` that combines `translateX(var(--ph-toggle-x))` (the
     position) with `scale(0.86)` (the press) — uses a CSS custom property
     so the inline-positioned translate doesn't conflict with the press scale.
   - Added `disabled` prop with the same opacity/cursor treatment.
3. `Slider`:
   - Thumb scales to 1.18× on hover with a 4px accent-tint halo; scales to
     0.92× on active (80ms) for a tactile grab; focus-within shows a stronger
     accent-tint-2 halo for keyboard users.
   - Added `disabled` prop; cursor-not-allowed; halo/scale rules skip when
     disabled via the `.ph-slider--disabled` class guard.
   - Moved the visual rules into `tokens.css` (`.ph-slider`, `.ph-slider__thumb`)
     so the JSX stays declarative.

**Why these matter:** PhButton, Toggle, and Slider are used in every settings
panel, the command palette, and most overlays. A 5ms tactile tweak compounds
into hundreds of perceived-quality wins across the app — and every animation
inherits the global `prefers-reduced-motion` clamp from iter 2, so the polish
is accessible by default.

**Next likely iteration target:** PhInput focus/validation polish — soft accent
glow on focus is already present, but invalid input has no visual treatment,
and the inputs in the modes/providers panels don't telegraph save-on-blur
intent. Or: streaming-text microinteractions in the command palette result
area (typing indicator, progressive shimmer) — the spec singles these out.

## Phase 2 — Native multi-window behavior

A new Ralph loop started here with a different ask: make the app actually feel
like a native Windows utility instead of a web page that visually mimics a
window. Specifically: real native overlay windows for things like a mode-switch
HUD that pops in transparently and fades away (Windows volume OSD style).

### Iter 6 (Phase 2 iter 1) — Native transparent mode-switch HUD window

**Problem found:** Every screen in the app is just a route inside the single
main webview window. The HUD-style "mode switched" toast the user asked for
isn't a real OS window — there's no separate native surface, no transparency,
no always-on-top behavior. The app reads like a tabbed website even when it
visually resembles a desktop tool.

**Fix — first real second window:**
1. `tauri.conf.json` — main window gets an explicit `"label": "main"` and a
   sensible 1100×760 default. Added a second window definition `mode-hud`:
   `decorations: false`, `transparent: true`, `alwaysOnTop: true`,
   `skipTaskbar: true`, `visible: false` at boot, `focus: false`, `shadow:
   false`, 320×88. This is a real OS window — no chrome, no taskbar entry,
   floats above other apps.
2. `capabilities/default.json` — `mode-hud` added to the windows allowlist so
   the HUD's React app can use `core:default` (event listening,
   `window.hide()`, etc.).
3. Backend `src-tauri/src/commands/overlay.rs` — new `show_mode_hud` command
   takes `{ modeId, modeName, iconName? }`. Looks up the `mode-hud` window,
   re-centers it horizontally on whichever monitor the user is currently on
   (DPI-aware via `current_monitor().scale_factor()`), positions it ~12% from
   the top (where Windows OSDs and Raycast confirmations sit), forces it
   always-on-top, shows it without stealing focus, then emits `hud_show` with
   the payload. Registered in `commands/mod.rs` and the `invoke_handler!`.
4. New feature `src/features/overlay-hud/` with `ModeHud` component that:
   - Overrides `<html>`/`<body>` background to transparent on mount (the
     default `var(--bg)` would otherwise paint the window solid).
   - Subscribes to `hud_show` via Tauri's `listen()`; on each event bumps a
     counter and stores the new payload.
   - Renders a glass card (heavy backdrop blur + saturate, soft border, accent
     glow) with a "MODE SWITCHED" eyebrow label and the new mode name.
   - Re-keys the card on every event so the CSS animation restarts on
     consecutive triggers.
   - After 1500ms calls `getCurrentWindow().hide()` so the window completely
     disappears (the React app stays alive in the background ready for the
     next event).
5. `src/app/App.tsx` — App is now **label-aware**. Reads
   `getCurrentWindow().label` once at module load, defaulting to `'main'`
   outside Tauri. If the label is `mode-hud`, the App renders only `<ModeHud />`
   — no `AppProviders` (Redux, TanStack Query, BrowserRouter), no
   `BackendThemeBridge`, no full router. Lightweight, fast, single-purpose.
6. `tokens.css` — new `ph-hud-popup` keyframes drive the full
   fade-in / hold / fade-out sequence in a single 1500ms animation
   (0→12%: fade in + scale up + drop in; 12%→82%: hold; 82%→100%:
   fade out + lift). Cubic-bezier 0.22,0.61,0.36,1 for a calm settle.
   Inherits the global `prefers-reduced-motion` clamp.
7. `HomePage.tsx` — added a "Test mode HUD" primary button that cycles
   through five preset modes (Developer → Email → Friendly → Concise →
   Documentation) so the user can repeatedly fire `show_mode_hud` and watch
   the real native window pop in and fade out. Catches and swallows the
   invoke error so the button is harmless when run in `vite dev` outside
   Tauri.

**Why this matters:** The HUD is the first real second native window in the
app. Once `npm run tauri dev` is running, clicking "Test mode HUD" produces an
actual OS-level translucent overlay positioned over whatever else is on
screen — exactly the Windows volume OSD / Raycast confirmation feel the user
asked for. The pattern (config new window → label-aware shell → backend
command that positions + shows + emits → React component that listens +
animates + hides) is now the template for every future native overlay (a
toast surface, a command-palette popover, future loading HUDs).

**Verification:** `cargo build` PASS, `cargo clippy -D warnings` clean,
`npm run build` PASS, all 38 Vitest tests still pass.

**Next iteration targets:**
1. Wire the HUD to fire automatically when the active prompt mode changes
   (currently emitted only by the test button). Needs a mode-change pathway —
   either a `set_active_mode` command, or hook into `mode_changed` event
   emission once sub-project 2 lands the AI engine.
2. Add a second native overlay window for toast notifications (similar
   pattern: separate transparent window, bottom-right corner, multi-stack).
3. Native window chrome polish on the main window — custom title bar, mica /
   acrylic backdrop, hide-on-close (minimize to tray) behavior. This pushes
   further into sub-project 3 (OS Integration) territory.
